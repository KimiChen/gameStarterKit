微信小游戏服务端 · 从「骨架正确」到「上线不丢钱」

你提的三段式（HTTPS 授权 → WS 网关取数/排位 → Colyseus 房间对战 + Redis 热缓存分钟落 MySQL）骨架是对的、也是主流打法。14 个 agent 逐子系统深化 + 对抗性评审后，真正的结论不是「7 份设计」，而是同一批失效模式在每个子系统反复出现——绝大多数指向「把强一致需求错误地建在 Redis 最终一致之上」。

架构总览

连接生命周期

wx.login() 拿 code → HTTPS /auth/wx-login 换 openid、签发存 Redis 的不透明 token（可撤销）→ WSS 带 token 连网关，onAuth 校验后 userId 由 token 反查（禁止客户端传 userId）→ 在共享 LobbyRoom 上走单一 rpc 通道取用户数据/排位 → 匹配后 join VersusRoom 对战 → 结算按数据分级落库。关键分野已经画进图里：货币走同步 MySQL 事务，进度/战绩走 Stream + 单写者 worker，局内易失态只留 Redis。

01 / The real findings

七个反复出现的失效模式

这是整轮评审最高价值的产出。下面每一条都在 3 个以上子系统被独立挖到——先建立这些横切原则，再看各子系统就都是它的实例。

P1

撤销/货币不能只押「删 Redis key」

critical ×6

Redis 主从异步复制 + AOF everysec：failover 会复活被删的 session、蒸发未落库的货币。封号后切到落后副本，被封会话又能查到；已 ack 的充值在复制跟上前主库崩溃即消失。真实 RPO = max(AOF fsync, 主从延迟)，高写入下是数秒，不是 1s。

Fix撤销引入与账号绑定的持久 token_epoch（落 MySQL），session 带签发时 epoch，落后即失效——「删 key」只是加速，不是唯一真相。货币彻底移出 Redis-then-batch：同步 write-through MySQL 事务（UPDATE balance + INSERT outbox 同事务），Redis 只做读缓存。
本轮修订：玩法的「撤销」不再是「恢复 MySQL 旧值」（MySQL 已无玩法数据）——只能前向补偿 clawback。

鉴权落库排行榜扩容房间横切
P2

noeviction + 热用户只进不出 = OOM 写雪崩

critical ×3

内存随「历史累计登录用户数」单调增长，而非在线数。数周后触顶，noeviction 下所有写（含新登录 SET、脏集合 SADD、续期 EXPIRE）一律 OOM 报错——全站写入雪崩，不是优雅降级。且「放独立逻辑库就不驱逐」是错的：maxmemory-policy 是实例级、逻辑库不隔离。

Fix冷用户降级 worker：已下线且脏集合为空者主动 DEL，权威回落 MySQL，下次 lazy 加载。物理隔离两类实例：durable（noeviction）与 cache（volatile-lru）。以 used_memory/maxmemory 0.7 为一级告警主动降级。
本轮修订（修法作废）：「冷用户降级到 MySQL + lazy 回源」不再可行——MySQL 无玩法权威数据，降无可降。新修法 = Redis-on-Flash（冷 value 下沉 SSD）+ 16384 桶分片扩容 + durable/cache 物理隔离。增长维度是累计注册用户数，不是在线数。

鉴权网关落库房间横切
P3

Stream 用 MAXLEN 裁剪 = 静默删掉未落库数据

critical ×3

多处设计以为 XADD MAXLEN ~ 只裁「已 ACK」的旧条目——错的。MAXLEN 按流长度无条件裁最老条目，完全不看消费组 PEL/ACK。worker 落后时正是积压最多、最容易触发的时刻，未落库的战绩/进度/货币被直接删除，违反「Stream 是唯一持久通道」的自定约束。

Fix改用基于已落库位点的 XTRIM MINID（只裁 < 全局已持久化 id 的条目）；MAXLEN 仅作远超峰值的兜底防爆；对 pending 深度 + 内存双告警 + 容量规划。裁剪必须是「已 ACK 之下」的不变量。
本轮修订（收窄）：玩法进度流已删，本条只对存活的可靠流生效：stream:match（P7 证据链）与 mail 必达流。

网关落库排行榜扩容房间
P4

幂等要「执行前占位」+ 下沉数据层

critical ×4

普遍是 GET 命中→否则执行→成功后 SET NX：GET 与执行之间没有原子占位，并发双发（双击/弱网重发/多开）双双 miss、双双执行 → 重复扣费/发奖。且「仅成功后 SET」不覆盖崩溃/超时；pending 用 24h 长 TTL 会变毒丸卡死用户 24 小时。

Fix执行前原子 SET NX PX 一个短租约（5–15s）哨兵占位，抢到才执行、完成覆写为结果；没抢到轮询/等结果。真正幂等下沉数据层：currency_ledger/purchases 用 UNIQUE(uid, idem_key)。key 绑 type + sha256(payload)，防跨操作串味。

网关落库房间横切
P5

at-least-once + HINCRBY 增量写 = 永久发散

critical ×3

游戏节点用非幂等 HINCRBY 直写 Redis 热态，MySQL 侧却用幂等语义。Stream 至少一次投递 + XAUTOCLAIM 重放一次：MySQL 不重复计、Redis 已多加且不回滚 → Redis 余额永远比 MySQL 多，且以谁为准无解。saveStore 又以 Redis 热态为权威。

Fix统一语义二选一：Redis 侧 HINCRBY 也用 matchId SETNX 守卫；或放弃游戏节点直写热态，由 flush worker 作唯一写者、以幂等方式同时更新 Redis 缓存与 MySQL（Stream 成唯一真源，Redis 降为物化视图）。回放按绝对值覆写而非重加 delta。补对账 job。
本轮修订（收窄）：玩法侧的「跨存储发散」按定义消失——只有一个存储，无副本可发散。对账只剩货币缓存一处；单写者 + 绝对值覆写保留，但目的从「防两库发散」变成「防单库重放双 apply」。

落库扩容房间
P6

fencing token 要守「业务写」，不只守租约行

critical ×1扩容主线

设计只对 singleton_lease 心跳行做 WHERE fencing_token<=$t，flush worker 批量写 wallet/progress 那一步没 token 守卫。GC 暂停超 TTL 的僵尸 leader 在自杀前已把过期批次写进业务表，新 leader 同时写 → 货币双写/覆盖。

Fix把「lease 守卫 UPDATE」与「业务批写」放进同一个 MySQL 事务，守卫 UPDATE 作第一句，受影响 0 行立即 ROLLBACK 并自杀。业务写绝不脱离 token 守卫单独提交。
本轮修订：玩法业务写现在落 Redis → fence 必须在 Lua 里对 hash 的 fence 字段做 CAS；MySQL WHERE last_fence <= :f 只保留给货币/订单。

扩容落库
P7

ranked 的「服务端权威」是假的

critical ×2

裁判模型下服务端只 mirror 客户端自报的 round/hearts/Defeated，不跑仿真；胜负 tiebreak 全靠客户端上报值。反作弊是纯事后软告警（console.warn + 落列），而金币/段位/ZSET/MySQL 战绩在校验前就已写死、无追回。天梯可被伪造进度和双开直接操纵。

Fixranked 结算把 gold/star/rank 落地延迟到无头重放校验通过之后（先写 pending、验过再 commit）；verdict=suspect 走 clawback：反向 XADD 冲销 delta + 从 ZSET 撤下。发奖类结算强制 100% 校验，抽样只用于低风险局。休闲局可保持乐观即时结算。
本轮修订：发奖被拆两半（货币走 MySQL、道具走 Redis）→ 单次发奖成为跨存储操作，必须走 outbox intent 协议；clawback 也分头：扣币走 MySQL 冲正、收道具走 Redis 补偿。

网关取数/排位数据层（借鉴 aisanguo-server）

先修正一个前提。aisanguo「重启不丢数据」不是因为数据同时在 Redis 和 MySQL。恰恰相反：它把 Redis 当作用户档的唯一真源，MySQL 只存 ServerUserModel 之类的汇总投影；load() 在 Redis miss 时直接判定「查无此人」，不回源 MySQL、没有重建逻辑。它的 durability 下限压在 Redis 自己的 RDB/AOF 上，不在应用代码里。

所以：借它的机制，反转它的持久化定位。它的模型逐条撞上我们的 P1/P2/P4/P5/P7。

它靠什么做到「重启不丢」——核心范式

一句话：进程不持有任何权威状态，进程只是无状态执行器。由三根支柱咬合而成：

  • 1数据落点固定：一个用户 = 一个 Redis Hash（HUser_<uid>），字段即 hash field。进程内存里的 PHP 对象只是投影，两级读缓存（loadedHashCache / permanentCache）重启即清空，下次 load(uid) 冷加载恢复。
    DiffHash.php:911-919 · RedisCacheTempData.php:8-14
  • 2写入边界 = 每个业务请求（action），不是定时快照。__lockUser(uid) 串行化 → HUser::load 从 Redis 拉 → __set 魔术方法触发字段级脏追踪 → 成功走 __commit（pipeline 只把本次脏字段 hMset 回去）／异常走 __rollback。

不可照搬（Node 模型不成立）

aisanguo 的 notice 是 Swoole 常驻多 worker + 协程，但数据层仍是「每 action share-nothing」。它的正确性建立在「uid 悲观锁把 action 串行 + 全局静态脏表每 action 边界清空 + swoole_atomic 单机锁」之上——这些前提在 Node 长驻单进程单线程 + async/await 在 await 点交错的模型下基本不复现。

✗1

进程级全局静态脏表 + __set 自动记脏

会串号

Node 单 loop 上并发 async handler 在 await 点交错：module 级可变脏表会在你的 await redis.hgetall 与 await redis.hmset 之间被另一个玩家的 RPC 改写 → 跨用户串脏，把 A 的改动 flush 进 B。RedisService::$changeInfos

Instead显式 unit-of-work：每个 RPC 建作用域 context 对象（绝不是单例）承载本次脏字段。用显式 set(field,val) API，不用 Proxy 魔术拦截，更不用 module 级全局收集。

✗2

Redis 是唯一真源、load miss 不回源

违背 P1/P2

这是与你的目标完全相反的持久化哲学。它 durability 押在 Redis RDB/AOF，MySQL 只是汇总投影，miss 即「查无此人」。

Instead反过来定位：MySQL = 货币/账号/订单的权威源与 durability 下限；Redis = 热态 write-through 缓存 + 排行等派生数据。Redis miss 必须回源 MySQL 重建，不是判定用户不存在。

✗3

先写 Redis 后写 MySQL、pipeline 当事务

货币蒸发

两者非同一事务、pipeline 非原子、事务 opt-in（默认 autocommit 逐条）。它承认崩溃会发散，但「MySQL 只是投影所以影响小」。对我们 MySQL 是货币权威源——这正是 P1 警告的失败路径。

InsteadMySQL 事务先落（真源），成功后才 write-through 更新 Redis 投影。多键原子用 Lua/MULTI，别拿 pipeline 当事务。

✗4

每请求 hGetAll 全量读回 + RMW

性能/竞态

PHP 一 action 一次全量往返可接受（且被 uid 锁串行）。Node 长连接高频 RPC 下：每 RPC 全量 hGetAll+悲观锁+commit 太重；read-modify-write 在两次 await 间开了竞态窗口。为省往返把对象常驻房间内存跨 RPC 复用，又把权威态漂回进程内存。

Instead读用字段级 HMGET，别整对象 hGetAll；能用服务端原子操作就不做 RMW；货币直接绕开 Redis-RMW 走 MySQL 事务。

✗5

swoole_atomic 单机锁 + 200ms 忙轮询

违背 P6

单机多 worker 有效、多机不跨机（它自己 gotcha 里承认），且无 fencing——锁只保证「拿到锁才执行」，不校验写入时锁是否仍有效。200ms 忙轮询在 Node 单 loop 上是纯空转。

Instead单进程内用 per-uid async mutex / promise 队列（await 排队，不轮询）；多网关实例用 Redis 锁 + fencing token 一路守到业务写（MySQL WHERE last_fence <= :f），不只守租约行。

✗6

排行纯 ZSET、无 MySQL 镜像、读榜即发奖

违背 P7

Redis 丢 = 榜和「谁能领奖」状态一起真丢、无法重建（它自己承认是弱项）。ActivityRankFinalize 直接读 ZSET 前 N 名即算奖励发放——把未经校验的分数当权威。

Instead榜是派生数据：定期把 top-N 快照进 MySQL rank_snapshot；发奖延迟到服务端重放校验通过才写 rank_award，异常走 clawback（负向 ledger 冲正）。

// 写：单一提交边界
async function withUser<T>(uid: string, fn: (uow: UnitOfWork) => Promise<T>) {
  return localMutex(uid, async () => {              // ① 进程内 async 队列，不轮询
    const lease = await acquireLease(uid);           // ② 跨实例 Redis 锁 + fence
    const uow = new UnitOfWork(uid, lease.fence);
    try   { const r = await fn(uow); await uow.commit(); return r; }
    catch (e) { uow.discard(); throw e; }             // 没 commit 就没写，天然不落
    finally { await lease.release(); }                // Lua CAS 校验 fence 再 DEL
  });
}

我在合成基础上改掉的 6 处

  • 1取数路径不该取分布式锁。原方案把只读的 user.getInfo 也包进 withUser，等于每次看档都抢一把跨实例锁。→ 读写分路，读走 readUser（read-through，无锁、无脏表）。
  • 2lock fence 与撤销 epoch 不能是同一个计数器。原方案「每次抢锁 INCR fence」又说「撤销会话 = INCR fence」——每个请求都在 bump，撤销语义就失效了。→ 拆成两个：last_fence（每次抢锁递增，守并发写）与 token_epoch（仅封号/踢人时递增，守鉴权，落 MySQL）。
  • 3INSERT IGNORE → ODKU id = id：前者会把截断/NOT NULL/CHECK 全降级为 warning 静默吞数据（见「MySQL 8.0 迁移」节）。
  • 4忙轮询 sleep(200) 递归抢锁（照抄 delayUserActionByLock）在 Node 上纯空转 loop、且无超时上界。→ 进程内 per-uid async mutex 排队；跨实例锁有界重试 + 超时。
  • 5锁 PX 5s 中途过期不需要看门狗续租——这正是 fencing 的意义：僵尸写会在 MySQL WHERE last_fence <= :f 处被拒。但 Redis 投影也要 Lua CAS 比 fence，否则旧 writer 能覆盖新值。
  • 6rebuildFromMysql 要 single-flight：灾后大量用户同时重连触发 lazy 回填 = 对 MySQL 的缓存击穿风暴（这也是横切节的遗漏项之一）。

✅ 本轮拍板：Redis 权威模型

三条决定（已定）：① 玩法状态（背包/进度/战力）真源 = Redis，MySQL 不是权威。② 重建源 = Redis 自身备份（RDB/AOF/主从/快照），与 MySQL 无关。③ RPO 按字段分级；容量走 Redis-on-Flash 分层。
拍板前先拆掉的一个矛盾：「玩法真源不在 MySQL」与「从 MySQL 完整重建 Redis」不能同时成立——MySQL 里没有玩法状态，就没有重建源。而且即便建了管道，运维也只能重建到最后一次成功 flush 的位点，Redis 领先的那段 delta（= flush 延迟）不可恢复，玩家会「刚打完的关回退、刚抽的卡蒸发」。

已定为：重建源是 Redis 自己。那句「从 MySQL 重建」作废，改为「运维用 Redis 原生持久化 + 备份保证不丢」，并须签 RPO/RTO 数字（见下）。

跨存储原子性：扣钱在 MySQL、发道具在 Redis

两个权威源之间不可能做 2PC。用 MySQL 作协调者 + outbox intent，Redis 作幂等下游副作用，靠崩溃重放收敛。

不变式

永不「扣了钱没发货」（intent 与扣钱同事务落盘，relayer 必补）· 永不「发了货没扣钱」（apply 严格在扣钱提交之后）。最坏只是延迟，绝不丢、不双花。

hash-tag 的硬约束：跨用户的全局 key（rank:{type}:{season}）与 user:{uid} 不同槽，不能塞进同一条 Lua。排行两段式 hydrate（多用户 HMGET rank_sub）、全服邮件批量发道具、公会战力汇总，在 Cluster 下全是 CROSSSLOT。要么逐个操作放弃跨用户原子性，要么 hash-tag 强绑同槽（引入热点倾斜）。Redis 形态（Sentinel 单主 vs Cluster）必须上线前拍板。

要运维同事签字的 SLA

「Redis 不会丢」在 appendfsync everysec + 异步主从下是假的：真实丢失 = max(1s fsync 窗口, 主从复制延迟)，failover 提升的是落后的异步副本，高写入时延迟可达数秒。而玩法态没有 ledger、无对账能事后发现。所以这些必须量化并签字：

  • 1持久化策略：durable 实例 Redis-on-Flash + appendfsync everysec。不给整实例上 always（写 QPS 会从十万级掉到几千~万级）——贵重效果由 MySQL outbox intent 兜底，而非靠 fsync。
  • 2failover RPO 上界数字：异步复制延迟的承诺值。若某些字段要 RPO=0，走 WAIT numreplicas 1 半同步（延迟翻倍，副本宕机时写阻塞）。
  • 3冷启动 RTO 实测：RPO=0 ≠ RTO=0。几十 GB 的 RDB 加载 / AOF 重放要几分钟到几十分钟，期间全站玩法读写不可用（不是降级，是全停）。
  • 4定期 DR 演练：真的拿备份恢复一次。很多团队的「能重建」只是纸面承诺，真出事才发现 AOF 损坏 / 恢复脚本跑不通。
  • 5durable(noeviction) 与 cache(allkeys-lru) 物理分实例。「放独立逻辑库 SELECT n 就能隔离内存策略」是技术错误——maxmemory-policy 是实例级，逻辑库共享同一 maxmemory。
  • 6大 key 运维：删除一律 UNLINK（同步 DEL 大 Hash 单条就能卡住实例几十毫秒）；--bigkeys 采样监控；RDB fork 预留 ~2× 内存（COW）。
最后一句提醒：Redis「物理不丢」≠「Redis 里的值是对的」。持久性可以外包给运维，一致性不行。P4（执行前幂等占位）、P6（fence 守业务写）、大 Hash HGETALL 阻塞单线程（禁止全量 load，一律 HMGET + 背包拆独立 key）、玩法 schema 演进没有 ALTER（靠 schemaVersion 双读 + 懒迁移 worker）——这些都是应用层责任，运维的物理持久性承诺一条也堵不住。
09 / Migration

MySQL 8.0 迁移映射

从 PostgreSQL 切 MySQL 8.0 不是换连接串——几处核心模式语义不同，还改变了两条原评审结论。下面是经一轮 convert→verify 校验后的要点。前置：生产 MySQL ≥ 8.0.19（吃满 ODKU 行别名 / CHECK 强制 / 函数索引 / 多值索引）。

切 MySQL 改变的两条评审结论

Δ1

「批内同一 uid 报错」在 MySQL 消失

语义变更

PG 的 cannot affect row a second time 在 MySQL 多值 ODKU 下不报错，按行顺序后者覆盖前者、静默。

Fix靠这个报错发现重复的逻辑要改成应用层去重，或用 version + GREATEST 让结果与批内顺序无关。

Δ2

分区约束打破 match_id 唯一性

must fix

分区键必须进每个唯一键，match_results PK 被迫改 (match_id, created_at) → 同一 match_id 配不同 created_at 能插多行，重复结算不再被 UNIQUE 挡住。这正是 P7/S6「matchId 幂等」依赖的唯一性。

Fix单独保 match_id 唯一：独立非分区索引表 match_index(match_id UNIQUE) 做幂等闸，或写入侧 ODKU/先 SELECT 显式排重。不能只靠分区表的联合唯一键。

MySQL 8.0 相对 PG 的新坑

  • crit默认 REPEATABLE READ + gap/next-key 锁：并发范围写、二级索引条件更新、outbox 取行的死锁（1213）/锁等待（1205）远多于 PG RC。→ 写路径一律主键等值定位；货币/转账/outbox 会话切 RC（需 binlog_format=ROW）；全局保留死锁重试 + 指数退避。
  • maj多步 DDL 不可整体回滚（单条 DDL 在 8.0 是原子/崩溃安全的，但 expand-contract 的多步不是一个事务）。大表 ALTER 用 gh-ost/pt-osc，小改用 ALGORITHM=INSTANT；每步幂等可重入。
  • majutf8mb4 默认排序不敏感撞键 + 索引键长 3072B 上限：id/token/hash/idem_key 用 ascii_bin；GET_LOCK 连接作用域易在连接池泄漏 → fencing 用 lease 表。
  • maj无 RETURNING、连接池下 LAST_INSERT_ID 连接局部：插入与取号放同连接/同事务，直接用 mysql2 result.insertId 而非另发 SELECT。
10 / Consistency

统一约定（跨子系统对齐）

评审暴露的一批问题源于各子系统各写各的。上线前把这些钉死，避免键冲突与原子性陷阱：

约定规则
Redis 实例划分durable（noeviction，AOF：session/dirty/Streams/presence/ZSET）与 cache（volatile-lru：热档只读缓存）物理分离。「逻辑库隔离内存策略」是错的，别用。
Cluster 与 hash-tag拓扑（Sentinel vs Cluster）现在就定，不能留 open decision——它决定多键 Lua/原子撤销能否成立。同实体键用 {uid} / {matchId} hash-tag 共 slot；排行按 (season,mode) 分实例。
时钟源所有令牌桶/租约/tie-break 一律用 Redis redis.call('TIME')，禁止 app 节点传入 nowMs（多节点漂移会污染）。
版本字段进度版本 ver 与货币版本 cver 分开；条件写各比各的；MySQL 侧 user_snapshot.version 与 user_currency.version 不混用同一列语义。
userId 真源一律从校验过的 token 反查，禁止客户端传 userId；friends/matchId/delta 一律服务端权威，忽略客户端列表。
Stream 裁剪只用 XTRIM MINID 按已落库位点裁；MAXLEN 仅远超峰值的兜底。消费一律 consumer group + XACK + XAUTOCLAIM 接管 pending。
货币边界已 ack 即不可丢：真金 100% 同步 MySQL 事务（UPDATE + INSERT outbox 同事务），Redis 只读缓存。RPO：货币=0，进度/战绩=flush 端到端延迟上界（明示、纳入 SLO）。
事务隔离MySQL 默认 REPEATABLE READ + 间隙锁，死锁面比 PG 的 RC 大。货币/转账/outbox 等高并发短事务会话切 READ COMMITTED（需 binlog_format=ROW）；所有写路径主键等值定位 + 死锁重试。
字符集/排序业务文本 utf8mb4_0900_ai_ci；id/token/idem_key/hash 用 ascii_bin——默认不敏感排序会让 u_Ab/u_ab 撞主键。

12 / Sequencing

建议落地顺序

不要一次上全套。按「先消除会丢钱/丢数据的路径，再铺扩容」推进：

  1. 前置实测

    验证两个致命假设

    真机实测 Colyseus 0.17 的 RedisDriver + 定向建房能力；确认货币同步 write-through 在目标充值 QPS 下延迟可接受。这两条不过关，拓扑和货币方案都要改。

  2. 地基

    SQLite → MySQL 8.0，双写过渡

    把 accounts / users / currency_ledger / outbox 建到 MySQL 8.0（≥ 8.0.19），save TEXT→JSON 校验入库；saveStore 从 SQLite 切过去，大表变更走 gh-ost。这是所有强一致路径的前提。

  3. 强一致优先

    货币纯同步 + 撤销 epoch

    货币 100% 走 UPDATE balance + INSERT outbox 同事务，删除任何 dirty:currency 后门；引入 MySQL token_epoch 让撤销不依赖删 Redis key。（P1）

  4. 鉴权

    不透明 token + onAuth

    替换 saveStore 的 deviceId 占位；wx-login 查 status、refresh 原子轮换 + grace、去掉 deviceId 降级后门。

  5. 落库层（已瘦身）

    删管道 + gameplay_outbox relayer

    删掉脏集合 / RENAME 快照 / user_snapshot / flush worker 定时通道 / 玩法侧 version 对账。新建 gameplay_outbox + relayer 承接「扣钱(MySQL) + 发道具(Redis)」的跨存储原子性；stream:match 保留为 P7 证据链、按 MINID 裁剪。（P3/P4/P5/P6 均已收窄或改法）

  6. 对局正确性

    matchId 前移 + 本地兜底 + ranked 校验闸

    matchId 开局生成写 state；结算先落本地持久队列；ranked 发奖延迟到无头重放通过后 + clawback。（P7）

  7. 容量先行

    Redis-on-Flash + 16384 桶分片路由

    玩法真源在 Redis 且只进不出，内存随累计注册用户线性增长——冷用户无处可降级（P2 修法已作废）。先上 on-flash 分层与桶路由表（加实例只搬部分桶），durable/cache 物理分实例。同时向运维取得 RPO/RTO 签字与一次真实 DR 演练。

  8. 扩容

    presence/driver + proxy + drain

    seat-aware proxy；per-node drain gate；dead-node reaper。先拍板 Sentinel vs Cluster（CROSSSLOT 影响排行 hydrate 与全服发奖）。最后才铺水平扩容。
